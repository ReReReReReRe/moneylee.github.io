原生JDBC访问数据库缺点和ORM框架介绍

  • 原生jdbc缺点：
    • 每次要加载驱动，连接信息是硬编码
    • sql和java代码耦合在一起，改动不易
    • 参数设置缺少灵活性
    • 处理结果麻烦。不能自动映射到实体类
    • 连接资源不能复用，资源浪费
  • 什么是ORM框架
    • 对数据库的表和POJO对象做映射的框架
  • 市面上的ORM框架
    • hibernate
      • 比较笨重，sql调优麻烦
      • 不推荐
    • JPA - Spring Data JPA
      • Java persistrnce API，轻量级，部分中小项目适合
      • 次推荐
    • mybatis
      • 半自动化（半ORM框架），便于写sql，轻量级，广泛使用
      • 推荐

新版ORM框架Mybatis3.x基础知识

  • 什么是Mybatis
    • Apahce的一个开源项目
    • 一款优秀的持久层框架，它支持自定义sql，存储过程及高级映射
    • 免除了几乎所有JDBC代码以及设置参数和获取结果集的工作
    • 通过简单的xml或注解来配置和映射Java对象到数据库中的记录
  • 核心流程：
    • 每个基于Mybatis的应用都是一个以一个 SqseesionFactory的实例为核心
    • SqseesionFactory的实例可以通过SqseesionFactoryBuilder获得
    • SqseesionFactoryBuilder可以从XML配置文件或一个预先配置的Configuration实例来构建出SqseesionFactory实例
    • 工厂设计模式里面需要获取Sqseesion，里面提供了在数据库执行SQL命令所需的方法

Mybatis3.x 复杂sql查询

  • Mybatis的SQL语句返回结果有两种
    • resultType
      • 查询出的字段在相应的pojo中必须有和它相同的字段对应，或者基本数据类型
      • 适合简单查询
    • resultMap
      • 需要自定义字段，或者多表查询，比resultType更强大
      • 适合复杂查询

ResultMap复杂对象一对多查询结果映射之collection

  • collection：一对多查询结果映射，比如user有多个订单

Mybatis3.x多级缓存和懒加载

  • 什么是缓存
    • 程序经常要调用的对象在内存中，方便其使用时可以快速调用，不必去数据库或者其它持久化设备中查询，主要就是提高性能
  • Mybatis一级缓存
    • 简介：一级缓存的作用域是SQLSession，同一个SQLSession中执行相同的SQL查询（相同的SQL和参数），第一次回去查询数据库并写在缓存中，第二次会直接从缓存内读取
    • 基于PerprtualCache的HashMap本地缓存
    • 默认开启一级缓存
  • 失效策略：当执行SQL时 两次查询中间发生了增删改查的操作，即insert、update、delete等操作commit后会清空该SQLSession缓存

Mybatis3.x二级缓存的使用(上)

  • Mybatis二级缓存：
    • 简介：二级缓存是namespace级别的，多个SqlSession去操作同一个namespace下的Mapper的sql语句，多个SqlSession可以共用二级缓存，如果两个mapper的namespace相同，（即使是两个mapper，那么这两个mapper中执行sql查询到的数据也将存在相同的二级缓存区域中，但是最后是每个mapper的单独的名称空间）
    • 基于PerpetualCache的HashMap本地缓存,可自定义存储源,如Ecache/Redis等
    • 默认是没有开启二级缓存
    • 操作流程：第一次调用某个namespace下的SQL去查询信息，查询到的信息会存放在该mapper对应的二级缓存区域，第二次调用同个namespace下的mapper映射文件中，相同的sql去查询信息，会去对应的二级缓存区域去取
  • 失效策略：执行同个namespace下的mapper映射文件中增删改查sql，并执行了commit操作，会清空该二级缓存
  • 注意：实现二级缓存的时候，Mybatis建议返回的POJO是可序列化的，也就是建议实现Serializable接口
  • 缓存淘汰策略：会使用默认的LRU算法来收回（最近最少使用的）
  • 如何实现某个二级缓存mapper.xml里面配置

Mybatis3.X 懒加载

  • 什么是懒加载: 按需加载，先从单表查询，需要时再从关联表去关联查询，能⼤大提⾼数据库性能,并不是所有场景下使⽤用懒加载都能提⾼效率
  • Mybatis懒加载: resultMap里面的association、collection有延迟加载功能
  • dubug模式测试懒加载不准确，可以直接run

MyBatis3.X整合Mysql数据库事务

讲解Mybatis的事务管理形式

  • 使⽤用JDBC的事务管理

    • 使⽤用 java.sql.Connection对象完成对事务的提交(commit())、回滚(rollback())、关闭

      (close())

  • 使⽤用MANAGED的事务管理

  • 事务工厂TransactionFactory的两个实现类
    • JdbcTransactionFactory -> JdbcTransaction
    • ManagedTransactionFactory -> ManagedTransaction
  • 注意：如果不是web程序，然后使用的事务管理形式是MANAGED，那么将没有事务管理能力

Mysql的Innodb和MyISAM引擎的区别

简介:讲解mysql常⻅见的两种存储引擎的区别

区别项 Innodb myisam
事务 支持 不支持
锁粒度 行锁，适合⾼高并发 表锁，不适合高并发
是否默认 默认 非默认
支持外键 支持外键 不⽀持
适合场景 读写均衡,写⼤大于读场景，需要事务 读多写少场景，不需要事务
全⽂文索引 可以通过插件实现, 更更多使⽤用ElasticSearch 支持全⽂文索引

重点:MyISAM不不⽀支持事务，如果需要事务则改为innodb引擎 更改数据库的表里面的引擎

  • 为什么原先没进行commit操作，也可以插入成功?

    • 因为原先是myisam引擎,没有事务，直接插入成功

  • 检查数据库的 引擎 ，改为innodb

    • 多个表video/chapter/episode/user/video_order
    • video_banner

  • 案例例

  • 事务管理形式 MANAGED，设置⾮非⾃自动提交，然后注释 commit, 依旧可以保存成功

  • 不用重点关注，公司开发项⽬的事务控制基本是交给Spring，或者使用分布式事务
